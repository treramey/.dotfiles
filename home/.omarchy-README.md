# Omarchy-Specific Overrides

This directory contains platform-specific configuration overrides for Arch Linux / Omarchy machines.

## How It Works

The `.omarchy/` directory is nested within the config tree where overrides are needed. Files placed in `.omarchy/` override files in the parent directory when on an omarchy machine.

### Detection

The `dot` script checks if `$HOSTNAME` contains "omarchy" or "arch". If detected, it will:
1. Stow the base `home/` configuration first
2. Then scan for `.omarchy/` directories within `home/`
3. Override any matching files with their `.omarchy/` counterparts

### Directory Structure

Place `.omarchy/` directories inside config folders where you need overrides:

```
home/.config/
├── test/
│   ├── file.sh              # Base file (used on all machines)
│   └── .omarchy/
│       └── file.sh          # Omarchy override
└── nvim/
    ├── init.lua             # Base config
    └── .omarchy/
        └── init.lua         # Omarchy-specific nvim config
```

### Usage Examples

#### Example 1: Different shell config
```bash
# Create base file
mkdir -p home/.config/fish
echo "# Base fish config" > home/.config/fish/config.fish

# Create omarchy override
mkdir -p home/.config/fish/.omarchy
echo "# Omarchy fish config with Arch-specific paths" > home/.config/fish/.omarchy/config.fish
```

#### Example 2: Application-specific overrides
```bash
# Create base and override
mkdir -p home/.config/nvim/.omarchy
cp home/.config/nvim/init.lua home/.config/nvim/.omarchy/init.lua
# Edit the omarchy version with Linux-specific settings
```

#### Example 3: Multiple overrides in one directory
```
home/.config/app/
├── config.toml
├── settings.json
└── .omarchy/
    ├── config.toml    # Override config.toml
    └── settings.json  # Override settings.json
```

### Adding Overrides

1. Create the base file in `home/` (e.g., `home/.config/app/config.conf`)
2. Create the `.omarchy/` subdirectory (e.g., `home/.config/app/.omarchy/`)
3. Copy the base file to `.omarchy/` and make your omarchy-specific changes
4. Run `./dot stow` to apply

### Matching Rules

- Only files with matching names are overridden
- Files in `.omarchy/` that don't have a base counterpart are skipped
- Empty `.omarchy/` directories are ignored
- The override preserves the directory structure (symlinks point to the correct location)

### Backup Behavior

When an override replaces an existing file, the original is backed up to:
`backups/YYYYMMDD_HHMMSS_omarchy/`

## Ignoring Directories (`.omarchy-ignore`)

Some directories are macOS-only and shouldn't be stowed at all on omarchy machines.
List them in `home/.omarchy-ignore` (paths relative to `home/`):

```
# macOS-only window management
.config/aerospace
.config/skhd
.config/karabiner
```

After stow runs, any symlinks created for these paths are removed and empty directories cleaned up.

### Notes

- Co-located pattern: Overrides live next to the files they override
- Easy to see what's customized on omarchy - just look for `.omarchy/` directories
- The base files in `home/` remain in git; these overrides are machine-specific
- You can have `.omarchy/` directories at any depth in the config tree
- Use `.omarchy-ignore` for entire directories that shouldn't exist on omarchy
