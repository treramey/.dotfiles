Completed audit of local extensions in `home/.pi/agent/extensions` against pi extension API changes. Findings:

- Confirmed breaking issue in `home/.pi/agent/extensions/answer.ts`: uses `ctx.modelRegistry.getApiKey(...)` at lines 325 and 636. That API was removed in pi v0.63.0 in favor of `getApiKeyAndHeaders(model)`.
- Minor stale API usage in `home/.pi/agent/extensions/web-tools/render.ts`: uses old keybinding id `expandTools` in `keyHint(...)`; current API uses namespaced ids like `app.tools.expand`.
- No usages of `new ModelRegistry(...)`, so the v0.64.0 changelog’s direct breaking change does not affect local extensions.
- No stale uses found for removed provenance fields (`location`, `extensionPath`, etc.), and custom tool renderers appear compatible with the v0.62 render semantics.
- Important environment note: the global pi runtime is `0.64.0`, but extension imports resolve to local `home/.pi/node_modules/@mariozechner/*` packages at `0.61.0`, creating version skew and increasing risk for runtime helper imports.

Recommended next steps:
1. Update `answer.ts` to use `getApiKeyAndHeaders()` and pass both `apiKey` and `headers` into `complete()`.
2. Change `web-tools/render.ts` to `keyHint("app.tools.expand", ...)`.
3. Align local `@mariozechner/*` packages under `home/.pi/node_modules` with the installed pi runtime version.
